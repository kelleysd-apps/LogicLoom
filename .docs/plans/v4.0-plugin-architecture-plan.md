## Executive Summary

Framework v4.0 represents a fundamental architectural shift: migrating from the current monolithic skills/agents/hooks/scripts structure into a **plugin-first architecture** leveraging Claude Code's native plugin system. This enables hot-swappable capability modules, simplified updates via RL feedback, and true multi-agent swarm orchestration.

## Part 3: Plugin Architecture Deep Research

### Claude Code Plugin System — Architecture Analysis

#### Plugin Structure (Official Standard)

```
plugin-name/
├── .claude-plugin/
│   └── plugin.json          # Required: Plugin manifest (name, version, metadata)
├── commands/                 # Slash commands (.md files) — auto-discovered
├── agents/                   # Subagent definitions (.md files) — auto-discovered
├── skills/                   # Agent skills (subdirs with SKILL.md) — auto-discovered
│   └── skill-name/
│       ├── SKILL.md          # Required per skill
│       ├── scripts/          # Helper scripts
│       ├── references/       # Reference docs
│       └── examples/         # Usage examples
├── hooks/
│   ├── hooks.json            # Event handler configuration
│   └── scripts/              # Hook implementation scripts
├── .mcp.json                 # MCP server definitions (optional)
└── README.md                 # Documentation
```

#### Key Plugin Capabilities

| Capability | Details |
|------------|---------|
| **Auto-discovery** | All `.md` files in `commands/`, `agents/`, skills subdirs scanned automatically |
| **Hot-swap** | `claude plugin enable/disable` without restart |
| **Versioning** | Per-plugin semantic versioning in plugin.json |
| **Hooks** | Full event system: PreToolUse, PostToolUse, Stop, SubagentStop, SessionStart, SessionEnd, UserPromptSubmit, PreCompact, Notification |
| **MCP bundling** | Plugins can bundle MCP servers that auto-start |
| **Portable paths** | `${CLAUDE_PLUGIN_ROOT}` for all internal references |
| **State management** | `.claude/plugin-name.local.md` pattern for per-project config |
| **Marketplace** | Install from GitHub-hosted marketplaces |
| **Python + Bash hooks** | Hooks can be Python or bash scripts |

#### Proven Multi-Agent Swarm Pattern (from official examples)

The **multi-agent-swarm** plugin pattern (documented in plugin-dev references) shows:

```yaml
# .claude/multi-agent-swarm.local.md
---
agent_name: auth-implementation
task_number: 3.5
pr_number: 1234
coordinator_session: team-leader
enabled: true
dependencies: ["Task 3.4"]
additional_instructions: "Use JWT tokens, not sessions"
---
# Task: Implement Authentication
Build JWT-based authentication...
```

This coordinates via:
- **tmux sessions** for agent-to-agent communication
- **Stop hooks** for coordinator notification when agents idle
- **State files** for task tracking and dependency management
- **Git worktrees** for parallel branch work

## Part 5: Agentic Swarm Architecture

### Swarm Orchestration Pattern

```
User: /swarm "Build user auth with React UI, Express API, PostgreSQL"
  │
  ├─ Orchestrator analyzes domains: [frontend, backend, database, security]
  │
  ├─ Phase 1: Sequential Dependencies
  │   └─ database-specialist → schema design (git worktree: feat/auth-schema)
  │
  ├─ Phase 2: Parallel Execution
  │   ├─ backend-architect → API + auth (git worktree: feat/auth-api)
  │   └─ frontend-specialist → React UI (git worktree: feat/auth-ui)
  │
  ├─ Phase 3: Integration
  │   └─ full-stack-developer → wire up frontend ↔ backend
  │
  ├─ Phase 4: Validation
  │   ├─ testing-specialist → E2E tests
  │   └─ security-specialist → auth review
  │
  └─ Coordinator merges PRs and reports results
```

### Implementation via CLI

```bash
# Coordinator spawns agent in separate terminal/tmux pane
claude --print \
  --agent database-specialist \
  --model opus \
  --fallback-model sonnet \
  --max-budget-usd 5.00 \
  --permission-mode delegate \
  --system-prompt "You are working on Task 3.1: Design auth schema..." \
  "Design the PostgreSQL schema for user authentication..."
```

### Swarm State Management

Each swarm agent gets a `.claude/multi-agent-swarm.local.md`:
```yaml
---
agent_name: backend-api-auth
task_number: 3.2
coordinator_session: auth-swarm-leader
enabled: true
dependencies: ["3.1"]
budget_usd: 5.00
worktree: feat/auth-api
---
# Task: Implement Auth API
...
```

### Swarm Coordination via Stop Hooks

When an agent finishes (Stop event), the hook:
1. Notifies coordinator via tmux
2. Updates task status
3. Triggers dependent agents

## Part 6: Benefits Analysis — Plugin Migration

### Why Plugin-First Architecture Would Work

| Benefit | Details |
|---------|---------|
| **Hot-swap skills** | `claude plugin disable sdd-domain-frontend` — instant, no restart |
| **Independent versioning** | Update `sdd-debug` to 2.0 without touching governance |
| **RL per plugin** | Track success/failure rates per plugin, update individually |
| **Selective installation** | Backend-only project? Skip frontend/design plugins |
| **Marketplace distribution** | `claude plugin install sdd-governance@sdd-marketplace` |
| **Reduced complexity** | Each plugin is self-contained: skills + agents + hooks + scripts |
| **Easier testing** | Test each plugin in isolation |
| **Community plugins** | Third-party plugins can extend framework |
| **Automatic updates** | `claude plugin update sdd-debug` — no git pull needed |

### Risks and Concerns

| Risk | Mitigation |
|------|------------|
| Plugin system is new (v2.1.32) — may have bugs | Start with governance plugin, validate stability |
| Loss of monolithic coherence | `sdd-governance` plugin enforces cross-plugin compliance |
| Hook conflicts between plugins | Document hook priorities, use matchers carefully |
| Constitution enforcement across plugins | Governance plugin hooks run on ALL events, regardless of other plugins |
| Token overhead if too many plugins loaded | Progressive disclosure already limits token usage |
| Migration effort is significant | Phased approach — convert one plugin at a time |

### RL Integration with Plugins

```
Plugin Execution → Hook captures success/failure → RL metrics update
                                                       ↓
                            Plugin selection_weight adjusted
                                                       ↓
                            Hot-swap underperforming plugins
                            with improved versions
```

Each plugin's `plugin.json` can include RL metadata:
```json
{
  "name": "sdd-domain-backend",
  "version": "1.2.0",
  "rl_metrics": {
    "success_rate": 0.85,
    "selection_weight": 0.9,
    "last_evaluated": "2026-02-06"
  }
}
```

## Part 9: Plugin System Deep Research (Added 2026-02-06)

### Claude Code Plugin Architecture — Full Capabilities Map

Based on deep analysis of the official marketplace (28 internal plugins, 13 external plugins):

#### Plugin Component Types

| Component | Format | Auto-Discovery | Can Bundle |
|-----------|--------|----------------|------------|
| **Commands** | `.md` in `commands/` | ✅ Yes | Slash commands (/name) |
| **Agents** | `.md` in `agents/` | ✅ Yes | Subagent definitions |
| **Skills** | `SKILL.md` in `skills/*/` | ✅ Yes | Procedural knowledge |
| **Hooks** | `hooks.json` in `hooks/` | ✅ Yes | Event handlers (9 event types) |
| **MCP Servers** | `.mcp.json` at root | ✅ Yes | External tool integrations |
| **Scripts** | Any in `scripts/` | Manual | Helper utilities |
| **State** | `.claude/plugin-name.local.md` | Manual | Per-project config |

#### Hook Event Types Available

| Event | When | Use Case |
|-------|------|----------|
| `PreToolUse` | Before any tool executes | Git safety gates, input validation |
| `PostToolUse` | After tool completes | RL metric capture, logging |
| `Stop` | Agent wants to stop | Swarm coordination, loop control |
| `SubagentStop` | Subagent finishes | Swarm task completion signals |
| `SessionStart` | Session begins | Context loading, state initialization |
| `SessionEnd` | Session ends | Cleanup, metric finalization |
| `UserPromptSubmit` | User sends message | Governance preflight |
| `PreCompact` | Before context compaction | State preservation |
| `Notification` | System notification | Alert handling |

#### Proven Multi-Agent Patterns from Official Plugins

**1. Parallel Agent Spawning** (`feature-dev` plugin):
- Spawns 2-3 `code-explorer` agents in parallel
- Then 2-3 `code-architect` agents in parallel
- Then 3 `code-reviewer` agents in parallel
- Each agent uses `model: sonnet` for cost efficiency
- Results merged by orchestrating command

**2. Autonomous Agent Loop** (`ralph-loop` plugin):
- Stop hook blocks session exit
- Reads last output, checks for completion promise
- If not done: feeds prompt back, increments iteration
- Capped at max_iterations for safety

**3. Swarm Coordination** (`multi-agent-swarm` pattern):
- Each agent gets `.claude/multi-agent-swarm.local.md` state file
- Stop hook notifies coordinator via tmux when agent idles
- State tracks: agent_name, task_number, pr_number, dependencies
- Coordinator manages task graph and dependency resolution

**4. Dynamic Hook Rules** (`hookify` plugin):
- Users create `.claude/hookify.*.local.md` rule files
- Plugin hooks scan for rule files at runtime
- Rules can be enabled/disabled without restart
- Supports: warn (allow but notify) and block (prevent execution)

### Plugin vs Current Architecture — Detailed Comparison

| Aspect | Current (Monolithic) | Plugin-First |
|--------|---------------------|-------------|
| **Adding a skill** | Create SKILL.md + update skill-index.json + update routing | Create SKILL.md in plugin dir (auto-discovered) |
| **Adding an agent** | Create .md + update AGENTS.md + update CLAUDE.md | Create .md in plugin agents/ (auto-discovered) |
| **Adding a command** | Create skill + hook + document | Create .md in plugin commands/ (auto-discovered) |
| **Adding a hook** | Edit settings.json manually | Add to plugin hooks.json (auto-registers) |
| **Removing capability** | Edit multiple files | `claude plugin disable plugin-name` |
| **Updating one skill** | Risk breaking others | Update plugin version independently |
| **Distribution** | git clone + manual setup | `claude plugin install name@marketplace` |
| **Version tracking** | Single framework version | Per-plugin semantic versioning |
| **RL feedback** | Global skill-index.json | Per-plugin metrics in plugin.json |
| **Token efficiency** | All skills loaded always | Only enabled plugins load |
| **Testing** | Test entire framework | Test individual plugins in isolation |

### Key Finding: skill-index.json Elimination

The current `skill-index.json` (1,332 lines) serves three purposes:
1. **Skill registry** → Replaced by plugin auto-discovery
2. **Command routing** → Replaced by command `.md` files in `commands/`
3. **RL metrics** → Can live in per-plugin `plugin.json`

**This is the single biggest complexity reduction.** Instead of maintaining a centralized routing manifest, each plugin self-describes its capabilities through its directory structure and trigger descriptions.

### Feasibility Assessment

| Question | Answer | Evidence |
|----------|--------|----------|
| Can plugins define hooks? | ✅ Yes | hookify, ralph-loop, security-guidance all do |
| Can plugins bundle agents? | ✅ Yes | feature-dev (3 agents), agent-sdk-dev (2 agents) |
| Can plugins bundle skills? | ✅ Yes | plugin-dev (7 skills), hookify (1 skill) |
| Can plugins bundle MCP servers? | ✅ Yes | All external plugins do (supabase, firebase, etc.) |
| Can plugins use model overrides? | ✅ Yes | Commands support `model:` frontmatter |
| Can hooks block operations? | ✅ Yes | hookify demonstrates warn + block patterns |
| Can plugins coordinate with each other? | ✅ Via state files | multi-agent-swarm uses .local.md pattern |
| Are plugins hot-swappable? | ✅ Yes | `claude plugin enable/disable` without restart |
| Can we create our own marketplace? | ✅ Yes | Known marketplaces are GitHub repos |

### Risk: What Can't Plugins Do?

| Limitation | Impact | Workaround |
|------------|--------|------------|
| No plugin dependency declarations | Can't enforce "governance must be installed" | Document in README, check at runtime via hook |
| No hook priority ordering across plugins | Governance hook might fire after other hooks | Use UserPromptSubmit (fires before tool hooks) |
| No shared state between plugins | Can't directly access another plugin's internals | Use `.claude/*.local.md` convention |
| Plugin system is new (2.1.x) | May have bugs, API may change | Start with governance PoC, validate stability |

## Part 12: Initial Enhancement Analysis (from Session Start)

### Context

These enhancements were identified during the initial project analysis session before the deep plugin research was conducted. They represent the original feature wishlist that informed the v4.0 direction.

### Enhancement Category 1: Agent Team Templates

Pre-defined team compositions for common multi-agent workflows. Each team template defines:
- Which agents to spawn
- Execution mode (parallel vs sequential)
- Dependency ordering
- Budget allocation per agent

#### Proposed Templates

| Template | Agents | Execution | Use Case |
|----------|--------|-----------|----------|
| **research-team** | 3× parallel researchers + 1 synthesizer | Parallel → Sequential | Technology evaluation, architecture decisions |
| **build-team** | architect → implementor → reviewer | Sequential | Feature implementation |
| **review-team** | security + quality + performance reviewers → synthesizer | Parallel → Sequential | Pre-merge code review |
| **full-stack-team** | frontend + backend + database in parallel → integration specialist | Parallel → Sequential | Cross-domain feature development |

#### Implementation Path

In plugin architecture, team templates become commands in `sdd-orchestrator`:
```
sdd-orchestrator/
├── commands/
│   ├── swarm.md              # /swarm (custom team)
│   ├── research-team.md      # /research-team (pre-defined)
│   ├── build-team.md         # /build-team (pre-defined)
│   ├── review-team.md        # /review-team (pre-defined)
│   └── fullstack-team.md     # /fullstack-team (pre-defined)
├── agents/
│   ├── swarm-coordinator.md
│   └── team-synthesizer.md   # Merges parallel agent outputs
└── skills/
    └── team-orchestration/SKILL.md
```

### Enhancement Category 2: Swarm Cost Controls

Per-agent and per-team budget enforcement using `--max-budget-usd`:

```bash
# Team budget allocation example
TEAM_BUDGET=20.00
AGENT_BUDGET=$(echo "$TEAM_BUDGET / $AGENT_COUNT" | bc -l)

claude --print \
  --agent backend-architect \
  --max-budget-usd $AGENT_BUDGET \
  --fallback-model sonnet \
  "Design the auth API..."
```

Features:
- **Team-level budget cap**: Total spend limit across all swarm agents
- **Per-agent allocation**: Distribute budget proportionally or by priority
- **Automatic fallback**: `--fallback-model sonnet` when Opus quota depletes
- **Budget reporting**: Track spend per agent, per team, per session
- **Kill switch**: Terminate agents that exceed individual budget

### Enhancement Category 4: Dynamic Agent Registry

Currently, all agents are static `.md` files. With `--agents` CLI flag, agents can be defined at runtime:

```bash
claude --agents '{
  "auth-reviewer": {
    "description": "Reviews authentication implementations",
    "prompt": "You are a security specialist focused on auth..."
  },
  "schema-validator": {
    "description": "Validates database schema designs",
    "prompt": "You are a database architect who validates..."
  }
}' --print "Review this auth implementation..."
```

**v4.0 Benefits**:
- Swarm coordinator can **dynamically create specialized agents** based on task analysis
- No need to pre-define every possible agent variation as a static file
- Combine with `--system-prompt` for runtime personality injection
- Enables **task-specific agent variants** (e.g., "auth-focused backend-architect" vs general "backend-architect")

### Enhancement Category 6: RL-Driven Plugin Updates

With plugin-first architecture, RL feedback becomes actionable at the plugin level:

```
Execution Flow:
  User invokes /specify
  → sdd-specification plugin handles it
  → PostToolUse hook captures: success/failure, tokens, duration
  → RL metrics updated in plugin.json
  → If success_rate drops below threshold:
     → Auto-check for plugin updates: `claude plugin update sdd-specification`
     → Or flag for user: "sdd-specification success rate dropped to 72%. Update available (v1.3.0). Update? (y/n)"
```

This enables:
- **Automatic quality monitoring** per plugin
- **A/B testing**: Install v1.2 and v1.3 as separate plugins, compare RL metrics
- **Community quality signals**: Marketplace ratings based on aggregated RL data
- **Self-healing framework**: Underperforming plugins get updated or replaced
